SQL Basics & DDL / DML
The foundation — datatypes, constraints, CRUD operations, and query execution order
🔥 Always Asked SQL Query Execution Order Most common interview question #1 ▶
SQL is written in one order but executed in a completely different order. This explains why you can't use a SELECT alias in a WHERE clause.
Execution Order
-- Written order (how you type it):
SELECT  →  FROM  →  WHERE  →  GROUP BY  →  HAVING  →  ORDER BY  →  LIMIT

-- Actual execution order:
1. FROM      -- identify tables, apply JOINs
2. WHERE     -- filter rows (before grouping)
3. GROUP BY  -- group remaining rows
4. HAVING    -- filter groups (after grouping)
5. SELECT    -- choose columns, compute expressions
6. DISTINCT  -- remove duplicates
7. ORDER BY  -- sort result
8. LIMIT     -- restrict rows returned

-- Why this matters:
SELECT SUM(salary) AS total
FROM employees
WHERE total > 50000; -- ❌ ERROR: alias not yet defined at WHERE step

SELECT SUM(salary) AS total
FROM employees
HAVING SUM(salary) > 50000; -- ✅ HAVING runs after SELECT
Q: Why can't we use a SELECT alias in WHERE?
WHERE executes before SELECT, so the alias doesn't exist yet when WHERE is processed.
Q: What's the difference between WHERE and HAVING?
WHERE filters individual rows before grouping. HAVING filters groups after GROUP BY. You can't use aggregate functions in WHERE.
⬆ High DDL Commands — CREATE, ALTER, DROP, TRUNCATE Conceptual + syntax ▶
DDL (Data Definition Language) defines the structure of your database. Changes are auto-committed in most RDBMS.
DDL Examples
-- CREATE with all common constraints
CREATE TABLE employees (
  emp_id    INT          PRIMARY KEY AUTO_INCREMENT,
  name      VARCHAR(100) NOT NULL,
  email     VARCHAR(100) UNIQUE,
  dept_id   INT          REFERENCES departments(dept_id),
  salary    DECIMAL(10,2) CHECK (salary > 0),
  hire_date DATE         DEFAULT CURRENT_DATE
);

-- ALTER: add / modify / drop column
ALTER TABLE employees ADD COLUMN phone VARCHAR(15);
ALTER TABLE employees MODIFY COLUMN salary DECIMAL(12,2);
ALTER TABLE employees DROP COLUMN phone;

-- DROP vs TRUNCATE vs DELETE
DROP TABLE employees;          -- removes table + structure + data
TRUNCATE TABLE employees;     -- removes all data, keeps structure, faster
DELETE FROM employees;        -- removes data row-by-row, can be rolled back

⬆ High Constraints — PK, FK, UNIQUE, CHECK, NOT NULL Definition + differences ▶
Constraints enforce data integrity rules at the database level — not the application level.
Primary Key vs Unique Key?
PK: uniquely identifies a row, cannot be NULL, only one per table. UNIQUE: ensures no duplicate values, can have NULL (one NULL per column in most DB), multiple per table.
What is a Foreign Key?
A column (or set) that references the PRIMARY KEY of another table. Enforces referential integrity — you can't insert a value that doesn't exist in the referenced table.
ON DELETE CASCADE vs ON DELETE SET NULL?
CASCADE: deleting parent row also deletes all child rows. SET NULL: deleting parent sets FK column to NULL in child rows. RESTRICT: prevents delete if child rows exist.
Can a table have no Primary Key?
Yes, but it's bad practice. Without a PK, duplicate rows can exist and referential integrity can't be enforced.
🔥 Always Asked SELECT — Filtering with WHERE, LIKE, IN, BETWEEN Pattern matching + ranges ▶
WHERE Variants
-- Basic filtering
SELECT * FROM employees WHERE salary BETWEEN 50000 AND 100000;
SELECT * FROM employees WHERE dept_id IN (1, 2, 3);
SELECT * FROM employees WHERE manager_id IS NULL;

-- LIKE pattern matching
SELECT * FROM employees WHERE name LIKE 'A%';     -- starts with A
SELECT * FROM employees WHERE name LIKE '%son';   -- ends with son
SELECT * FROM employees WHERE name LIKE '_a%';    -- 2nd char is 'a'

-- NULL comparisons (== never works with NULL)
SELECT * FROM employees WHERE manager_id = NULL;   -- ❌ Returns nothing!
SELECT * FROM employees WHERE manager_id IS NULL; -- ✅ Correct

-- DISTINCT
SELECT DISTINCT dept_id FROM employees; -- unique dept IDs only
● Medium DML — INSERT, UPDATE, DELETE Data manipulation ▶
DML Syntax
-- INSERT single row
INSERT INTO employees (name, dept_id, salary)
VALUES ('Alice', 1, 75000);

-- INSERT multiple rows
INSERT INTO employees (name, dept_id, salary) VALUES
  ('Bob', 2, 60000),
  ('Carol', 1, 80000);

-- INSERT from SELECT
INSERT INTO archive_employees
SELECT * FROM employees WHERE hire_date < '2020-01-01';

-- UPDATE with WHERE (always use WHERE!)
UPDATE employees
SET salary = salary * 1.10
WHERE dept_id = 1;

-- DELETE with WHERE
DELETE FROM employees
WHERE emp_id = 42;
What happens if you run UPDATE without WHERE?
Every row in the table gets updated. This is a very common destructive mistake. Always use WHERE with UPDATE and DELETE.
⬆ High NULL Handling — COALESCE, NULLIF, IS NULL Tricky in interviews ▶
NULL means "unknown" — not zero, not empty string. Any comparison with NULL returns NULL (not TRUE or FALSE), which is why = NULL never works.
NULL Functions
-- COALESCE: returns first non-NULL value
SELECT COALESCE(phone, email, 'no contact') AS contact
FROM employees;

-- NULLIF: returns NULL if both args are equal, else first arg
SELECT NULLIF(score, 0) FROM results; -- turns 0 into NULL

-- Safe division (avoid divide by zero)
SELECT revenue / NULLIF(units, 0) AS avg_price FROM sales;

-- COUNT behavior with NULL
SELECT
  COUNT(*)           -- counts all rows including NULLs
  COUNT(manager_id) -- counts only non-NULL values
FROM employees;
⬆ High CASE WHEN — Conditional Logic in SQL Used in almost every real query ▶
CASE Expressions
-- Simple CASE
SELECT name,
  CASE dept_id
    WHEN 1 THEN 'Engineering'
    WHEN 2 THEN 'Marketing'
    ELSE 'Other'
  END AS department
FROM employees;

-- Searched CASE (more flexible)
SELECT name, salary,
  CASE
    WHEN salary >= 100000 THEN 'Senior'
    WHEN salary >= 60000  THEN 'Mid'
    ELSE 'Junior'
  END AS level
FROM employees;

-- CASE inside aggregate (pivot-style)
SELECT
  SUM(CASE WHEN dept_id = 1 THEN salary ELSE 0 END) AS eng_total,
  SUM(CASE WHEN dept_id = 2 THEN salary ELSE 0 END) AS mkt_total
FROM employees;

JOINs In Depth
The most tested SQL topic — know every type and when to use each
🔥 Always Asked All JOIN Types with Visual Explanation #1 most tested SQL concept ▶

JOIN Examples
-- INNER JOIN: employees who have a department
SELECT e.name, d.dept_name
FROM employees e
INNER JOIN departments d ON e.dept_id = d.dept_id;

-- LEFT JOIN: all employees, even those without a department
SELECT e.name, d.dept_name
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.dept_id;

-- Find employees with NO department (LEFT JOIN anti-pattern)
SELECT e.name
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.dept_id
WHERE d.dept_id IS NULL;  -- rows with no match

-- SELF JOIN: find manager name for each employee
SELECT e.name AS employee, m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.emp_id;
⬆ High Multiple JOINs & Chaining Tables Real-world queries ▶
3-Table JOIN
-- employees → departments → locations
SELECT
  e.name        AS employee,
  d.dept_name   AS department,
  l.city        AS location
FROM employees e
JOIN departments d  ON e.dept_id   = d.dept_id
JOIN locations l    ON d.location_id = l.location_id
WHERE l.country = 'India'
ORDER BY d.dept_name;
⬆ High Set Operations — UNION, INTERSECT, EXCEPT Combine result sets ▶
Set operations combine results from two SELECT statements. Both queries must return the same number of columns with compatible data types.
Set Operations
-- UNION: all rows from both (removes duplicates)
SELECT name FROM employees
UNION
SELECT name FROM contractors;

-- UNION ALL: keeps duplicates (faster)
SELECT name FROM employees
UNION ALL
SELECT name FROM contractors;

-- INTERSECT: rows in both results
SELECT product_id FROM orders_2023
INTERSECT
SELECT product_id FROM orders_2024;

-- EXCEPT/MINUS: rows in first but not second
SELECT customer_id FROM all_customers
EXCEPT
SELECT customer_id FROM active_customers;

Aggregation & Grouping
COUNT, SUM, AVG, GROUP BY, HAVING — the core of data analysis queries
🔥 Always Asked GROUP BY + HAVING — Grouping with Filters Core aggregation pattern ▶
Aggregation Queries
-- Department summary
SELECT
  dept_id,
  COUNT(*)         AS headcount,
  AVG(salary)      AS avg_salary,
  MAX(salary)      AS max_salary,
  SUM(salary)      AS total_cost
FROM employees
GROUP BY dept_id;

-- HAVING: only departments with > 5 employees
SELECT dept_id, COUNT(*) AS headcount
FROM employees
GROUP BY dept_id
HAVING COUNT(*) > 5
ORDER BY headcount DESC;

-- Combined WHERE + GROUP BY + HAVING
SELECT dept_id, AVG(salary) AS avg_sal
FROM employees
WHERE hire_date >= '2020-01-01'   -- filter rows first
GROUP BY dept_id
HAVING AVG(salary) > 70000;      -- filter groups after
Rule: what columns can appear in SELECT with GROUP BY?
Only columns that are in the GROUP BY clause, or are inside an aggregate function (COUNT, SUM, etc.). Any other column is ambiguous.
● Medium ROLLUP, CUBE, GROUPING SETS Advanced grouping ▶
Extensions to GROUP BY that compute subtotals and grand totals in a single query.
ROLLUP / CUBE
-- ROLLUP: subtotals + grand total (hierarchical)
SELECT dept_id, job_title, SUM(salary)
FROM employees
GROUP BY ROLLUP(dept_id, job_title);
-- produces: (dept, job), (dept, NULL), (NULL, NULL)

-- CUBE: all combinations of subtotals
SELECT dept_id, year, SUM(salary)
FROM employees
GROUP BY CUBE(dept_id, year);
-- produces: (dept,yr), (dept,NULL), (NULL,yr), (NULL,NULL)

Subqueries & CTEs
Nested logic, correlated queries, and the WITH clause
🔥 Always Asked Subquery Types — Scalar, Inline, Correlated Must understand all 3 ▶
Subquery Examples
-- 1. Scalar subquery: returns single value
SELECT name, salary
FROM employees
WHERE salary > (SELECT AVG(salary) FROM employees);

-- 2. Inline view (derived table): subquery in FROM
SELECT d.dept_name, t.avg_sal
FROM (
  SELECT dept_id, AVG(salary) AS avg_sal
  FROM employees
  GROUP BY dept_id
) t
JOIN departments d ON t.dept_id = d.dept_id;

-- 3. Correlated subquery: references outer query
SELECT e.name, e.salary
FROM employees e
WHERE e.salary = (
  SELECT MAX(salary)
  FROM employees
  WHERE dept_id = e.dept_id  -- references outer e.dept_id
);
-- Runs once per outer row → can be slow on large tables

-- EXISTS vs IN (EXISTS stops at first match)
SELECT name FROM employees e
WHERE EXISTS (
  SELECT 1 FROM orders o
  WHERE o.emp_id = e.emp_id
);
EXISTS vs IN — which is faster?
EXISTS short-circuits on first match and doesn't materialize the subquery result. IN evaluates all values. For large datasets, EXISTS is generally faster, especially with correlated queries.
⬆ High CTEs — WITH Clause & Recursive CTEs Modern SQL must-know ▶
CTEs make complex queries readable by naming intermediate results. Recursive CTEs handle hierarchical data like org charts and trees.
CTE Examples
-- Simple CTE: reusable named subquery
WITH dept_avg AS (
  SELECT dept_id, AVG(salary) AS avg_sal
  FROM employees
  GROUP BY dept_id
)
SELECT e.name, e.salary, da.avg_sal
FROM employees e
JOIN dept_avg da ON e.dept_id = da.dept_id
WHERE e.salary > da.avg_sal;

-- Recursive CTE: traverse org chart hierarchy
WITH RECURSIVE org_tree AS (
  -- Base case: top-level managers
  SELECT emp_id, name, manager_id, 0 AS level
  FROM employees
  WHERE manager_id IS NULL

  UNION ALL

  -- Recursive case: subordinates
  SELECT e.emp_id, e.name, e.manager_id, ot.level + 1
  FROM employees e
  JOIN org_tree ot ON e.manager_id = ot.emp_id
)
SELECT * FROM org_tree ORDER BY level;

Window Functions
ROW_NUMBER, RANK, DENSE_RANK, LAG, LEAD, running totals — the most powerful SQL feature
🔥 Always Asked ROW_NUMBER, RANK, DENSE_RANK — The Big 3 Most tested window function ▶
Window functions compute a value for each row based on a "window" of related rows, without collapsing them like GROUP BY does. They always use the OVER() clause.
FunctionTies get same rank?Gap after tie?Example (scores: 100,100,90)
ROW_NUMBER()❌ (arbitrary)N/A1, 2, 3
RANK()✅ Same rank✅ Yes1, 1, 3
DENSE_RANK()✅ Same rank❌ No gap1, 1, 2
Ranking Functions
SELECT
  name, dept_id, salary,
  ROW_NUMBER() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rn,
  RANK()       OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rnk,
  DENSE_RANK() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS drnk
FROM employees;

-- Classic interview: Top N per group (e.g. top 3 salaries per dept)
SELECT * FROM (
  SELECT name, dept_id, salary,
    DENSE_RANK() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS dr
  FROM employees
) ranked
WHERE dr <= 3;  -- top 3 per department
⬆ High LAG() & LEAD() — Access Adjacent Rows Time-series & comparison queries ▶
LAG / LEAD
-- LAG: value from previous row
-- LEAD: value from next row
SELECT
  month,
  revenue,
  LAG(revenue, 1, 0) OVER (ORDER BY month) AS prev_revenue,
  revenue - LAG(revenue) OVER (ORDER BY month) AS growth
FROM monthly_sales;

-- Detect consecutive login (common interview problem)
SELECT user_id, login_date,
  LAG(login_date) OVER (PARTITION BY user_id ORDER BY login_date) AS prev_login
FROM logins;
⬆ High Running Totals & Moving Averages Frame specification ▶
Window Frames
-- Running total (cumulative sum)
SELECT order_date, amount,
  SUM(amount) OVER (
    ORDER BY order_date
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS running_total
FROM orders;

-- 7-day moving average
SELECT date, revenue,
  AVG(revenue) OVER (
    ORDER BY date
    ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
  ) AS moving_avg_7d
FROM daily_sales;

-- FIRST_VALUE / LAST_VALUE per partition
SELECT name, dept_id, salary,
  FIRST_VALUE(salary) OVER (PARTITION BY dept_id ORDER BY salary DESC) AS max_in_dept
FROM employees;
● Medium NTILE, PERCENT_RANK, CUME_DIST Distribution analysis ▶
Distribution Functions
-- NTILE: divide rows into N equal buckets
SELECT name, salary,
  NTILE(4) OVER (ORDER BY salary) AS quartile
FROM employees;

-- PERCENT_RANK: relative rank as 0 to 1
SELECT name, salary,
  PERCENT_RANK() OVER (ORDER BY salary) AS pct_rank
FROM employees;

Indexes & Performance
Why queries are slow, how indexes work, and how to use EXPLAIN
🔥 Always Asked Index Types & When Indexes Are Used Performance interview staple ▶
An index is a separate data structure (usually a B-Tree) that allows the DB to find rows without scanning the full table. It speeds up reads but slows down writes.
Creating & Using Indexes
-- Simple index
CREATE INDEX idx_emp_dept ON employees(dept_id);

-- Composite index (column order matters!)
CREATE INDEX idx_dept_salary ON employees(dept_id, salary);
-- Useful for: WHERE dept_id = 1 AND salary > 50000
-- Also useful for: WHERE dept_id = 1 (leftmost prefix rule)
-- NOT useful for: WHERE salary > 50000 alone

-- Unique index
CREATE UNIQUE INDEX idx_email ON employees(email);

-- EXPLAIN: view query execution plan
EXPLAIN SELECT * FROM employees WHERE dept_id = 1;
-- Look for: type=ALL (full scan, bad) vs type=ref/index (good)
When does MySQL NOT use an index?
When using functions on indexed column (WHERE UPPER(name)='X'), using LIKE with leading wildcard ('%abc'), when cardinality is low (boolean column), or when the query returns >30% of rows (full scan is cheaper).
Clustered vs Non-Clustered index?
Clustered: rows are physically stored in index order. Only one per table (usually PK). Non-clustered: separate structure pointing to row locations. Multiple allowed per table.

Transactions & ACID
Atomicity, isolation levels, and concurrency problems
🔥 Always Asked Transaction Control — COMMIT, ROLLBACK, SAVEPOINT ACID + isolation ▶
Transaction Syntax
-- Basic transaction
BEGIN;
  UPDATE accounts SET balance = balance - 500 WHERE id = 1;
  UPDATE accounts SET balance = balance + 500 WHERE id = 2;
COMMIT;  -- both succeed

-- Rollback on error
BEGIN;
  DELETE FROM orders WHERE status = 'cancelled';
  -- something fails…
ROLLBACK;  -- undo everything in this transaction

-- SAVEPOINT: partial rollback
BEGIN;
  INSERT INTO log VALUES ('step1');
  SAVEPOINT sp1;
  INSERT INTO log VALUES ('step2');
  ROLLBACK TO sp1;  -- undoes step2 only
COMMIT;  -- commits step1
What are the 4 ACID properties?
Atomicity (all or nothing), Consistency (valid state before/after), Isolation (concurrent txns don't interfere), Durability (committed data survives crash).
Dirty read vs Non-repeatable read vs Phantom read?
Dirty read: reading uncommitted data from another transaction. Non-repeatable: same SELECT returns different values in same transaction. Phantom: same range query returns different rows (rows inserted by another transaction).

Normalization
1NF → 2NF → 3NF → BCNF — the theory behind good schema design
🔥 Always Asked 1NF → 2NF → 3NF → BCNF with Examples Theory + practical conversion ▶
1NF (First Normal Form)
Each cell must have a single atomic value. No repeating groups or arrays. Every row must be uniquely identifiable (have a PK). Example violation: a "phones" column storing "9876, 8765".
2NF (Second Normal Form)
Must be in 1NF + no partial dependency. Every non-key column must depend on the ENTIRE primary key (only applies when PK is composite). Example: if PK is (student_id, course_id), storing student_name (which depends only on student_id) violates 2NF.
3NF (Third Normal Form)
Must be in 2NF + no transitive dependency. Non-key columns must not depend on other non-key columns. Example: emp_id → dept_id → dept_name. dept_name transitively depends on emp_id via dept_id — violates 3NF.
BCNF (Boyce-Codd Normal Form)
Stronger than 3NF. For every functional dependency X → Y, X must be a super key. 3NF allows non-superkey determinants if Y is part of a candidate key. BCNF does not.

Must-Solve Query Problems
The 12 most commonly asked SQL interview problems — with solutions
🔥 #1 Asked Find the Nth Highest Salary Asked in 90% of interviews ▶
3 Approaches
-- Approach 1: DENSE_RANK (best, handles ties correctly)
SELECT salary FROM (
  SELECT salary,
    DENSE_RANK() OVER (ORDER BY salary DESC) AS dr
  FROM employees
) t
WHERE dr = 2;  -- change to Nth

-- Approach 2: Subquery (classic)
SELECT MAX(salary) FROM employees
WHERE salary < (SELECT MAX(salary) FROM employees);

-- Approach 3: LIMIT + OFFSET (MySQL, for Nth)
SELECT DISTINCT salary FROM employees
ORDER BY salary DESC
LIMIT 1 OFFSET 1;  -- OFFSET N-1 for Nth highest
🔥 Very Common Find and Delete Duplicate Rows Classic interview problem ▶
Duplicate Handling
-- Find duplicates
SELECT email, COUNT(*) AS cnt
FROM employees
GROUP BY email
HAVING COUNT(*) > 1;

-- Delete duplicates, keep row with lowest id
DELETE FROM employees
WHERE id NOT IN (
  SELECT MIN(id)
  FROM employees
  GROUP BY email
);

-- Using ROW_NUMBER (works in most databases)
DELETE FROM employees
WHERE id IN (
  SELECT id FROM (
    SELECT id,
      ROW_NUMBER() OVER (PARTITION BY email ORDER BY id) AS rn
    FROM employees
  ) t WHERE rn > 1
);
🔥 Very Common Employees Earning More Than Their Manager Self-join classic ▶
Self Join Solution
SELECT e.name AS employee, e.salary AS emp_salary,
       m.name AS manager,  m.salary AS mgr_salary
FROM employees e
JOIN employees m ON e.manager_id = m.emp_id
WHERE e.salary > m.salary;
🔥 Very Common Highest Salary Per Department Multiple valid approaches ▶
3 Approaches
-- Approach 1: Subquery
SELECT * FROM employees e
WHERE salary = (
  SELECT MAX(salary)
  FROM employees
  WHERE dept_id = e.dept_id
);

-- Approach 2: GROUP BY + JOIN
SELECT e.*
FROM employees e
JOIN (
  SELECT dept_id, MAX(salary) AS max_sal
  FROM employees
  GROUP BY dept_id
) mx ON e.dept_id = mx.dept_id AND e.salary = mx.max_sal;

-- Approach 3: Window function (cleanest)
SELECT * FROM (
  SELECT *,
    RANK() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rnk
  FROM employees
) t WHERE rnk = 1;
⬆ High Consecutive Rows / Streaks Tricky pattern ▶
Consecutive Detection
-- Find consecutive numbers appearing 3+ times
SELECT DISTINCT l1.num
FROM logs l1
JOIN logs l2 ON l1.id + 1 = l2.id AND l1.num = l2.num
JOIN logs l3 ON l2.id + 1 = l3.id AND l2.num = l3.num;

-- User login streak (consecutive days)
SELECT user_id, COUNT(*) AS streak
FROM (
  SELECT user_id, login_date,
    login_date - INTERVAL
      ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY login_date)
    DAY AS grp
  FROM logins
) t
GROUP BY user_id, grp
ORDER BY streak DESC;
⬆ High Month-over-Month Revenue Growth Data analyst favorite ▶
MoM Growth Query
SELECT
  month,
  revenue,
  LAG(revenue) OVER (ORDER BY month) AS prev_revenue,
  ROUND(
    (revenue - LAG(revenue) OVER (ORDER BY month))
    * 100.0
    / NULLIF(LAG(revenue) OVER (ORDER BY month), 0)
  , 2) AS growth_pct
FROM monthly_revenue
ORDER BY month;
⬆ High Pivot Table (Rows to Columns) Reporting queries ▶
Dynamic Pivot with CASE
-- Sales by quarter (rows → columns)
SELECT
  product_id,
  SUM(CASE WHEN quarter = 'Q1' THEN sales ELSE 0 END) AS Q1,
  SUM(CASE WHEN quarter = 'Q2' THEN sales ELSE 0 END) AS Q2,
  SUM(CASE WHEN quarter = 'Q3' THEN sales ELSE 0 END) AS Q3,
  SUM(CASE WHEN quarter = 'Q4' THEN sales ELSE 0 END) AS Q4
FROM product_sales
GROUP BY product_id;
⬆ High Customers Who Never Placed an Order Anti-join pattern ▶
3 Ways to Anti-Join
-- Method 1: LEFT JOIN + IS NULL (most efficient)
SELECT c.customer_id, c.name
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
WHERE o.customer_id IS NULL;

-- Method 2: NOT IN (careful with NULLs!)
SELECT customer_id, name
FROM customers
WHERE customer_id NOT IN (
  SELECT customer_id FROM orders
  WHERE customer_id IS NOT NULL  -- ⚠️ must exclude NULLs!
);

-- Method 3: NOT EXISTS (safe with NULLs)
SELECT customer_id, name
FROM customers c
WHERE NOT EXISTS (
  SELECT 1 FROM orders o
  WHERE o.customer_id = c.customer_id
);
● Medium String & Date Functions Commonly tested in written rounds ▶
Built-in Functions
-- String functions
SELECT
  CONCAT(first_name, ' ', last_name)  AS full_name,
  LENGTH(name)                         AS name_len,
  UPPER(name),  LOWER(name),
  TRIM(name),                           -- remove whitespace
  SUBSTRING(name, 1, 3),               -- first 3 chars
  REPLACE(name, 'a', '@'),
  INSTR(email, '@')                    -- position of @
FROM employees;

-- Date functions
SELECT
  NOW(),  CURDATE(),  CURTIME(),
  YEAR(hire_date),  MONTH(hire_date),  DAY(hire_date),
  DATEDIFF(NOW(), hire_date)           AS days_employed,
  DATE_ADD(hire_date, INTERVAL 30 DAY) AS probation_end,
  DATE_FORMAT(hire_date, '%Y-%m')    AS year_month
FROM employees;
